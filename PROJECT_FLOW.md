# TVC 短视频分镜项目流程

## 2026-05-15 流程更新

- 2026-05-29 首帧审查更新：分镜首帧图片全部完成后，必须逐张对照分镜表审查。除刻意模糊的人脸外，只要画面与分镜描述存在任何一处可见不符，就判定该图不可用，必须重新生成；不能用“基本接近”通过。
- 首帧重跑替换规则：重跑某一镜时可以先生成临时候选用于检查；一旦候选通过，自动覆盖原来的 `shot-XX-*.png` 稳定文件名，并让 `storyboard.html` 继续引用原文件名。不要把 `-v2`、`-v3` 等候选图作为最终交付资产保留，除非用户明确要求保留版本。
- 工作流拆成 3 个 project skills：`short-video-storyboard` 负责分镜脚本和连续性检查；`storyboard-html-delivery` 负责最终 HTML；`storyboard-first-frame-images` 只在用户确认后负责首帧图片。
- 分镜任务不再默认自动生成首帧图：用户要求“写分镜 / 出分镜 / TVC脚本 / 镜头设计”时，先输出中文分镜，但不执行图片生成。
- HTML 必须最终输出：无论用户是否需要首帧图片，每个分镜项目都要生成 `outputs/YYYY-MM-DD-<project-name>/storyboard.html`。未生成图片时，不展示图片卡片。
- 首帧图片需要二次确认：分镜和 HTML 完成后，直接在会话里询问用户是否需要首帧图，提供“需要生成首帧图 / 暂不生成首帧图 / 先修改分镜再决定”三个选项。不要使用 HTML 首帧确认面板。只有用户确认“需要生成首帧图”后，才执行 `storyboard-first-frame-images`。
- 首帧图输出位置：确认出图后，默认保存到当前项目 `outputs/YYYY-MM-DD-<project-name>/`，日期使用产出当天的北京时间日期，例如 `outputs/2026-05-15-string-lights/`。AIG image prompt 属于内部执行信息，默认不保存 `prompts/` 文件夹，也不在最终答复展示；只有用户明确要求保留或导出提示词时才保存。临时 prompt 文件、dry-run 文本、草稿 `.txt`、以及 assistant-only 工作笔记不得残留在日期交付目录；如重试出图时必须临时记录，应放在交付目录外或在交付前清理。
- 首帧图片参考规则：确认出图后，先生成并检查 `reference-character-3view-face.png` 与 `reference-scene-concept.png`。之后按 Shot 1 -> Shot 2 -> Shot 3 -> Shot 4 顺序生成每个首帧，但每个 shot 的图像输入只能是这两张参考图；不要把任何已生成的 `shot-*.png` 作为后续镜头的图像参考。镜头连续性通过提示词、分镜因果、稳定服装/道具和场景布局维护。
- 参考图状态完整性规则：`reference-scene-concept.png` 必须覆盖分镜中会反复出现、且影响因果的产品/空间状态，例如关灯/开灯、开/关、干/湿、安装前/安装后、触发前/触发后。状态不全时先补参考图，不要直接生成后续 shot。
- 场景参考机位规则：`reference-scene-concept.png` 是无人环境参考图，默认不要出现人物、人物剪影或脸。它需要尽量包含同一主场景或产品区域的全景/全局、中景、近景三类机位，帮助后续 shot 保持空间、产品位置和景别连续。
- 参考图锁定规则：一旦用户认可某张 `reference-*.png`，不要因为助手自己的主观担心而覆盖它。需要优化时先生成临时候选供检查，只有用户接受或它与明确需求存在客观不符时，才替换稳定文件名。
- 角色身份表物品清单规则：如果同一物品在 2 个及以上分镜中出现，`reference-character-3view-face.png` 的提示词必须加入紧凑物品清单，锁定每个重复物品的颜色、材质、大小、形状、佩戴/携带位置，以及由哪个角色使用或携带。
- 人物母版硬性约束：`reference-character-3view-face.png` 的提示词必须明确避免亚洲面孔，所有连续出现的人类角色不得呈现亚洲脸部特征。
- 物理世界连续性规则：设计分镜时必须检查天气、时间、空间、人物、道具、光线、事件因果是否连续。每个镜头新增的元素必须有来源：要么前面已经埋过，要么由当前动作产生，要么符合场景常识。不能为了硬塞卖点突然加入无因果的新状态，例如前 3 镜是干燥庭院，第 4 镜不能突然出现水滴；如果要表现防水，必须从前面就设定雨后、喷淋、泳池边、浇花或其他合理水源。
- 卖点植入优先级：当卖点证明和画面连续性冲突时，优先保证连续性，再换卖点表达方式。可以把该卖点移到另一个脚本版本，或在前序镜头补足因果；不要用不属于当前世界状态的画面强行证明功能。
- 短视频节奏规则：分镜不能像长视频截取片段。每个镜头必须一出现就带来新信息，并让观众在前 0.5 秒看出它在卖什么。少写人物完整走位和关系铺垫，多写触发点和结果瞬间：触发前的需求、产品介入的瞬间、人物/环境的结果反应、产品与场景绑定的记忆点。
- 权限规则：图片生成不能绕过系统审批；首帧图片的实际生成/编辑使用 `aig-image-gen` skill 与其 AIG CLI，必要时只请求窄范围可复用批准。
- 按钮式需求确认：本项目已提供本地 HTML 面板 `outputs/storyboard-brief-selector.html`。用户想点击选择时，打开该文件，填写基础信息，并点选核心卖点、场景、开头钩子、内容结构、留存节奏、转场技巧、拍摄技巧、声音与包装、平台、视频时长，复制右侧简报发回会话。面板默认面向社媒短视频创意，要求先给内容钩子、节奏、转场和拍摄剪辑建议，再输出分镜。视频时长按钮包含 15 秒、20 秒、30 秒、60 秒，默认 15 秒。
- 按钮面板复用规则：`outputs/storyboard-brief-selector.html` 是前置需求确认入口；首帧确认改为会话内完成，不再使用 HTML 面板。不要为每个视频项目复制或新建确认页面。每个视频项目默认只新建自己的日期项目目录，如 `outputs/YYYY-MM-DD-<project-name>/`。
- 分镜脚本格式规则：`storyboard.md` 默认使用 `outputs/2026-06-03-downlight-city-hero/storyboard.md` 的逐镜头块格式，不再使用宽 Markdown 表格。`## 分镜表` 下每个镜头按固定字段顺序书写：`镜头 N`、`时长：`、`景别：`、`运镜：`、`运镜逻辑：`、`画面描述：`、`文案旁白：`、`音效配乐：`。`时长` 写累计时间段，例如 `0-2.5秒`、`2.5-5秒`、`5-8秒`，不要写成单镜 `2.5 秒`。不要加入 `执行备注`、`首帧图方向`、`首帧方向`。

## 目标

把“产品/品牌信息”快速转化为可提案、可拍摄、可交给 AI 视频工具执行的短视频分镜。默认适用于 YouTube/TVC/Shorts/Reels 品牌推广片；未指定时默认 15 秒、3 个分镜，也支持 20 秒、30 秒、60 秒。镜头时长按广告功能、信息量和平台节奏灵活分配，可低于 5 秒。首帧图片必须二次确认后才生成。无论是否出图，都要有 HTML 分镜交付页。AIG image prompt 作为内部执行信息，不在默认交付物中展示。

## 标准流程

1. 需求收集
   - 如果用户只给了产品名，先启动提问机制，不直接生成完整分镜，不创建项目输出目录，不生成 HTML；必须等待用户补充简报，除非用户明确要求“按默认信息先写”。
   - 必问 6 项：产品/品牌、社媒内容钩子、内容结构或一句话大纲、核心卖点、场景设定、必须出现画面。
   - 明确产品/品牌、核心卖点、视频平台、比例和时长；目标受众只在用户主动提供或会影响画面时补充。
   - 收集风格偏好、竞品参考、禁用表达、品牌色、包装、场景或演员限制；logo、slogan、CTA 或合规限制只有用户主动提供时才加入。

2. 创意定调
   - 用一句话写清创意概念。
   - 确定情绪曲线：痛点/期待 -> 行动 -> 体验 -> 释放/记忆。
   - 选择社媒创意策略：开头钩子、留存节奏、转场技巧、拍摄技巧、声音与包装等。

3. 分镜设计
   - 按用户选择的视频时长安排镜头数量和节奏；镜头时长可长可短，取决于广告功能、信息量和平台节奏。未指定时默认 15 秒、3 个镜头；也可以根据创意密度调整镜头数量。
   - 每个镜头必须包含时长、景别、运镜、运镜逻辑、画面描述、文案旁白、音效配乐。
   - `storyboard.md` 的镜头内容使用逐镜头块：先写 `镜头 1`，再逐行写 `时长：`、`景别：`、`运镜：`、`运镜逻辑：`、`画面描述：`、`文案旁白：`、`音效配乐：`；后续镜头依次重复。`时长` 必须是累计时间段，起点和终点连续并覆盖总视频时长。
   - 每个镜头先确定一个明确的广告功能，再写画面：1 秒内建立痛点/场景、立即展示核心卖点、人物反应证明卖点、产品记忆点/购买理由。说不出广告功能的镜头不要只用来铺情绪。
   - 短视频镜头不要写成“建立场景 -> 人走过来 -> 两人交流 -> 再看产品”的长视频叙事；应写成“触发前 -> 触发瞬间 -> 结果瞬间 -> 记忆瞬间”的高密度广告叙事。
   - 每个镜头开头就要在动作中：第 2 镜可以是餐巾已经被风掀起、女生发丝已经动了；第 3 镜可以是饮品已经递到手边、人物已经对视微笑；结尾可以是风扇灯已经是主体，而不是等待镜头慢慢上仰过去。
   - 人物动作只作为卖点载体。避免镜头主要内容只是“人物走过去”“看见对方”“慢慢靠近”；优先写“一进入风场，衣角、发丝、餐巾、冰饮水汽等立刻出现可见变化”。
   - 15 秒内不要承载过多人物关系。浪漫、亲密、家庭感或社交感只能服务卖点：微笑代表舒适，正常音量交流代表低噪，发丝/餐巾动代表风量，大家留在户外代表场景适配。
   - 用剪辑跳跃替代真实走位。只要空间、人物、道具、产品和光线连续，就可以直接切到“已经进入廊架下”“已经被清风触动后抬眼一笑”“已经递到手边”的压缩动作。
   - `文案旁白` 默认写英文旁白或英文人物对话，除非用户明确要求中文文案；分镜字段、创意方向和设计解析仍用中文。
   - 有声旁白必须在同一个 `文案旁白` 字段里写音色锚点，例如 `音色：... "English line"`；同一条旁白声线在全片重复同一音色描述，同一个人物的对话也保持同一人物声音设定。
   - `文案旁白` 默认只写旁白或人物对话，不写屏幕文案；只有用户明确要求屏显文字/字幕时才加入。
   - `音效配乐` 只写当前镜头实际需要听到、且服务画面主体表达的声音。保持一个字段，根据画面需要写几个类别就写几个，尽量精简：全景环境可只写 `环境声`；人物对话写 `人声`；动作或道具成为主体时写 `动作音效` / `道具声`；`BGM` 只在需要情绪铺底或节奏推进时写。没有出现的声音直接省略，不写“不加某声音 / 不添加某音效 / 不额外添加产品声”等反向说明。
   - 镜头设计遵循“触发前需求 -> 产品介入瞬间 -> 结果/反应证明 -> 产品或场景记忆”。环境建立也要带痛点或产品信息，不单独慢铺氛围。
   - 不在此阶段生成首帧图片。

4. 设计解析
   - 说明景别为什么这样递进。
   - 说明运镜如何服务情绪和卖点。
   - 说明分镜之间如何保持场景、人物、产品和灯光连续。
   - 检查天气、时间、空间、人物、道具、产品状态、光线和事件因果是否连续。
   - 检查文案是否能在对应时长内被看清或听清。

5. 执行化
   - 拍摄项目：补充道具、场地、演员、灯光、剪辑和字幕方式；logo 或 CTA 只在用户明确提供时处理。
   - HTML 交付：使用 `storyboard-html-delivery` 生成单文件 HTML，包含项目概览、分镜表和设计解析，不展示 AIG image prompt。即使不出首帧图，也必须交付 HTML。
   - 首帧确认：HTML 完成后，直接在会话里让用户选择“需要生成首帧图 / 暂不生成首帧图 / 先修改分镜再决定”，不要打开或生成 HTML 确认面板。
   - AI 图片项目：仅当用户确认需要首帧图时，使用 `storyboard-first-frame-images` 编排连续性，并调用 `aig-image-gen` 执行实际出图。先生成人物三视图/面部细节参考图和场景概念图；场景概念图必须是无人环境参考图，包含分镜依赖的关键产品/空间状态，例如关灯/开灯、开/关、干/湿、安装前/安装后、触发前/触发后，并尽量覆盖全景/全局、中景、近景三类机位。参考图一旦被用户认可即锁定，需要优化时先出临时候选，不要直接覆盖稳定文件。如果同一物品跨 2 个及以上分镜出现，先在角色身份表提示词里加入物品清单，锁定颜色、材质、大小、形状、佩戴/携带位置和角色归属；之后每个 shot 都只把这两张参考图传给 AIG，不传入任何已生成的 `shot-*.png`。每一步先确认画面与分镜因果、人物服装、道具和场景布局连贯，再继续下一张。
   - 首帧图片全部生成后，必须做最终图文审查：逐张对照分镜表检查场景、时间、空间、人物、服装、道具、产品可见性/状态、光线、动作、构图意图和事件因果。除刻意模糊的人脸外，任何一处与分镜描述不符都视为不可用并重跑。画面中的文字、疑似文字、界面残影或字形伪影本身不作为失败原因，只有它们违背分镜、遮挡必要信息或冲突用户明确要求时才需要处理。
   - 重跑图片时，先输出临时候选进行检查；候选通过后自动覆盖原 `shot-XX-*.png` 文件，并更新同一个 `storyboard.html`。最终交付目录不保留 `-v2`、`-v3` 等临时候选文件，除非用户明确要求保留。
   - AI 视频项目：为每个分镜补充 video prompt，写清主体、动作、场景、镜头运动、光线、风格、负面限制。
   - 客户提案：补充创意方向、核心卖点映射和品牌记忆点。

6. 质检
   - 所选时长内信息是否过载。
   - 是否每个镜头的时长匹配它的信息量和广告功能；短镜头要清楚，长镜头不能空转。
   - 是否每个镜头都有不同作用，而不是重复展示产品。
   - 是否每个镜头前 0.5 秒就能看出它在卖什么；如果不能，重写为触发点或结果瞬间。
   - 是否存在只拍人物走位、看见对方、慢慢靠近的长视频过程镜头；如果有，压缩为广告功能明确的动作结果。
   - 是否有清晰 hook 和最终品牌记忆。
   - 是否能从画面直接看出卖点，不依赖大段解释。
   - 分镜或首帧图片是否保持同一场景、同一人物、同一产品位置和一致光线逻辑。
   - 首帧图片是否逐张完全匹配对应分镜描述；只要有一处可见不符，是否已判定不可用并重新生成。
   - 是否存在无因果的新天气、新道具、新人物、新空间或新产品状态。
   - 是否满足平台比例、字幕安全区和禁用内容要求；logo、slogan、CTA 或合规限制只检查用户明确提供的部分。

## 交付标准

- 客户能看懂创意：一句话创意清楚，卖点对应画面明确。
- 摄制组能执行：镜头、动作、时长、构图和声音完整。
- 剪辑师能剪：节奏递进清楚，文案量适合所选视频时长。
- AI 图片/视频工具能生成：每个分镜具备主体、动作、场景、镜头、风格和连续性锚点。
- 客户能预览：HTML 页面结构清楚，分镜表和设计解析便于查看；只有生成首帧图片后才展示图片卡片。
## Story Core Delivery Note

- Storyboard delivery must include a compact `故事核心` summary in the conversation response, `storyboard.md`, and `storyboard.html`. The summary should explain the causal arc, selling-point proof, and ending/payoff before the full shot table.
